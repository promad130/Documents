# Coding and Debugging Guidelines for Labs and Practice {#coding-and-debugging-guidelines-for-labs-and-practice}

Please keep the following in mind while coding your solutions (in lab as well as while practicing):

1. Use print statements to trace values and understand what your code is actually doing.  
   In C++, use `cerr` for debugging since `cerr`	 output is not read by the autograder, so you can keep debug statements while testing.  
   In Python, use `print(..., file=sys.stderr)`.  
     
2. Write custom test cases, especially edge cases. Compile using `g++` and run your program manually by providing input yourself.

3. Isolate the issue by commenting out parts of the code and testing components independently:  
     
* Is your predicate function correct?  
* Are your `lo` and `hi` values in binary search valid?  
* Does each helper function behave as expected?  
    
4. Learn how to compile by yourself from the terminal, and do not rely only on the autograder tool or VS Code. The compiler often provides better and more detailed error messages.  
     
5. Be comfortable using the STL properly. Using `std::sort` (including with a custom comparator), reversing arrays, and similar operations using standard library APIs should be familiar to you by now.  
     
6. Use meaningful variable names. It is difficult to understand what variables like `x`, `y`, `z` mean in your logic.  
     
7. Format your code using `Ctrl + Shift + P → Format Document` in VS Code. Unformatted code is hard to debug.
