Schema & models

Everything lives in prisma/schema.prisma

schema.prismaAnatomy of a schema file
// 1. Generator — what to generate
generator client {
  provider = "prisma-client-js"
}

// 2. Datasource — your database
datasource db {
  provider = "postgresql"          // or "mysql" | "sqlite"
  url      = env("DATABASE_URL")
}

// 3. Model — maps to a DB table
model User {
  id        Int       @id @default(autoincrement())
  email     String    @unique
  name      String?              // ? = optional field
  createdAt DateTime  @default(now())
  role      Role      @default(USER)
  posts     Post[]               // relation field
}

// 4. Enum — constrained string values
enum Role {
  USER
  ADMIN
}
Field types
String Int Float Boolean DateTime Json

Append ? for optional, [] for list.
Key attributes
@id — primary key
@unique — unique constraint
@default() — default value
@relation() — foreign key link
@@index([]) — composite index
@map() — custom column name
Model example
model Post {
  id    Int @id ...
  title String
  body  String?
  userId Int
  user  User @relation(...)
}

Database migrations

Prisma Migrate tracks every schema change as a versioned SQL file

1
Edit your schema
Add a model, a field, or change a type in schema.prisma.
2
Run dev migrate
npx prisma migrate dev --name add_post_table
Creates prisma/migrations/…/migration.sql, applies it to dev DB, regenerates client.
3
Commit migration files
Migration SQL files go into version control — they're your DB change history.
generated migration.sql
-- CreateTable
CREATE TABLE "Post" (
    "id"     SERIAL  NOT NULL,
    "title"  TEXT    NOT NULL,
    "userId" INTEGER NOT NULL,
    CONSTRAINT "Post_pkey" PRIMARY KEY ("id")
);
ALTER TABLE "Post" ADD CONSTRAINT "Post_userId_fkey"
    FOREIGN KEY ("userId") REFERENCES "User"("id");
!
Never use migrate dev in production
It can reset the database. Use migrate deploy instead — it applies only the pending migrations safely.
CLIProduction deploy
# Apply all pending migrations (safe, no reset)
npx prisma migrate deploy

# Then regenerate the client
npx prisma generate
Typical CI/CD step
Run prisma migrate deploy as part of your deployment pipeline before starting the app server. The command is idempotent — already-applied migrations are skipped.
CLICommon commands
# Check migration status
npx prisma migrate status

# Reset dev DB (drops + re-applies all migrations)
npx prisma migrate reset

# Push schema directly (no migration file) — prototyping
npx prisma db push

# Introspect existing DB → generate schema.prisma
npx prisma db pull

# Open Prisma Studio GUI
npx prisma studio

Prisma Client queries

Type-safe query API — auto-generated from your schema

TypeScriptCreate · Read · Update · Delete
import { PrismaClient } from '@prisma/client'
const prisma = new PrismaClient()

// CREATE — returns the created record
const user = await prisma.user.create({
  data: { email: 'alice@prisma.io', name: 'Alice' }
})

// READ — find one
const found = await prisma.user.findUnique({
  where: { id: 1 }
})

// READ — find many
const all = await prisma.user.findMany()

// UPDATE
const updated = await prisma.user.update({
  where: { id: 1 },
  data:  { name: 'Alice B.' }
})

// DELETE
await prisma.user.delete({ where: { id: 1 } })

// UPSERT — create OR update
const upserted = await prisma.user.upsert({
  where:  { email: 'alice@prisma.io' },
  update: { name: 'Alice Updated' },
  create: { email: 'alice@prisma.io', name: 'Alice' }
})
TypeScriptFiltering, sorting & pagination
// WHERE operators
prisma.user.findMany({
  where: {
    email: { contains: '@prisma.io' },
    name:  { startsWith: 'A' },
    age:   { gte: 18, lt: 65 },
    active: true
  }
})

// AND / OR / NOT
prisma.user.findMany({
  where: {
    OR: [
      { email: { endsWith: '@gmail.com' } },
      { email: { endsWith: '@outlook.com' } }
    ]
  }
})

// SORT
prisma.post.findMany({
  orderBy: { createdAt: 'desc' }
})

// PAGINATION — cursor-based (efficient)
prisma.post.findMany({ skip: 0, take: 10 })

// SELECT — only specific fields
prisma.user.findMany({
  select: { id: true, email: true }
})
TypeScriptIncluding & nesting relations
// Include related records
const user = await prisma.user.findUnique({
  where:   { id: 1 },
  include: { posts: true }   // fetches all posts
})
// user.posts → Post[]

// Filter the included relation
prisma.user.findUnique({
  where:   { id: 1 },
  include: {
    posts: {
      where:   { published: true },
      orderBy: { createdAt: 'desc' },
      take: 5
    }
  }
})

// Nested create (User + Post in one query)
prisma.user.create({
  data: {
    email: 'bob@example.com',
    posts: {
      create: [{ title: 'Hello world' }]
    }
  }
})

// Select specific nested fields
prisma.user.findMany({
  select: {
    name: true,
    posts: { select: { title: true } }
  }
})
TypeScriptTransactions, aggregates, raw
// TRANSACTION — all-or-nothing
await prisma.$transaction([
  prisma.user.create({ data: { email: 'x@y.com' } }),
  prisma.post.create({ data: { title: 'First', userId: 1 } })
])

// Interactive transaction (use result of step 1 in step 2)
await prisma.$transaction(async (tx) => {
  const user = await tx.user.create({ data: { email: 'z@y.com' } })
  return tx.post.create({ data: { title: 'Hi', userId: user.id } })
})

// AGGREGATES
const stats = await prisma.post.aggregate({
  _count: true,
  _avg:   { views: true },
  _max:   { views: true }
})

// RAW SQL (escape hatch)
const rows = await prisma.$queryRaw`
  SELECT * FROM "User" WHERE id = ${1}
`

Relationships

Three patterns — each with schema & query examples

User
1  →  many
Post
One User can author many Posts. Each Post has exactly one author. The foreign key (userId) lives on the "many" side (Post).
schema.prisma
model User {
  id    Int    @id @default(autoincrement())
  email String @unique
  posts Post[]  // ← 1-to-many: list of Posts
}

model Post {
  id     Int    @id @default(autoincrement())
  title  String
  userId Int          // foreign key column
  user   User   @relation(fields: [userId], references: [id])
}
TypeScriptQueries
// User with all their posts
prisma.user.findUnique({
  where: { id: 1 },
  include: { posts: true }
})

// Create post and connect to user
prisma.post.create({
  data: { title: 'Hello', user: { connect: { id: 1 } } }
})

// Count a user's posts
prisma.user.findUnique({
  where: { id: 1 },
  include: { _count: { select: { posts: true } } }
})
Post
many ↔ many
Tag
A Post can have many Tags, and a Tag can belong to many Posts. Prisma auto-creates a hidden join table _PostToTag. Use an explicit join model when you need extra fields on the relation.
schema.prismaAuto join table
model Post {
  id   Int   @id @default(autoincrement())
  tags Tag[] // implicit many-to-many
}

model Tag {
  id    Int    @id @default(autoincrement())
  name  String @unique
  posts Post[] // Prisma creates _PostToTag table
}
TypeScript
// Post with its tags
prisma.post.findUnique({
  where: { id: 1 },
  include: { tags: true }
})

// Connect existing tags to a post
prisma.post.update({
  where: { id: 1 },
  data: {
    tags: { connect: [{ id: 2 }, { id: 5 }] }
  }
})
schema.prismaWith extra fields on the relation
model Post {
  id       Int         @id @default(autoincrement())
  postTags PostTag[]
}

model Tag {
  id       Int         @id @default(autoincrement())
  name     String      @unique
  postTags PostTag[]
}

// Explicit join model — can carry extra fields
model PostTag {
  postId    Int
  tagId     Int
  addedAt   DateTime @default(now())  // extra field
  post      Post     @relation(fields: [postId], references: [id])
  tag       Tag      @relation(fields: [tagId], references: [id])
  @@id([postId, tagId])  // composite PK
}
User
1 ↔ 1
Profile
Each User has at most one Profile. The foreign key goes on the child model (Profile). The field is marked @unique to enforce the one-to-one constraint.
schema.prisma
model User {
  id      Int      @id @default(autoincrement())
  email   String   @unique
  profile Profile? // optional — user may have no profile
}

model Profile {
  id     Int    @id @default(autoincrement())
  bio    String?
  userId Int    @unique  // @unique enforces 1-to-1
  user   User   @relation(fields: [userId], references: [id])
}
TypeScriptQueries
// Create user + profile together
prisma.user.create({
  data: {
    email: 'alice@example.com',
    profile: {
      create: { bio: 'Engineer & cat lover' }
    }
  }
})

// Fetch user with their profile
prisma.user.findUnique({
  where:   { id: 1 },
  include: { profile: true }
})

// Update profile via user
prisma.user.update({
  where: { id: 1 },
  data:  {
    profile: { update: { bio: 'Updated bio' } }
  }
})
